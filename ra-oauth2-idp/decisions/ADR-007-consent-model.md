# ADR-007: コンセントは「subject × client × scope 集合」単位で永続化する

## ステータス

採用

## コンテキスト

コンセント（同意）の粒度設計には複数の選択肢がある:

- **クライアントごと（最大粒度）**: ユーザーが特定のクライアントに対して 1 度同意したら、
  以降そのクライアントが要求するスコープは UI なしで通過
- **スコープごと**: 各スコープに対して独立に同意状態を持つ
- **subject × client × scope 集合**: クライアント × 「過去に同意したスコープ集合」を 1 レコード
- **対話ごと**: 認可ごとに必ず再表示

業界の主流は「subject × client × scope 集合」である（Google, Microsoft, Auth0 など）。

## 決定

`spec/consent.schema.json` に従い、コンセントは以下の単位で永続化する:

```text
sub × client_id → { scopes[], granted_at, expires_at }
```

挙動:

1. `/authorize` で要求された全 scope が、既存コンセントの scopes 集合に含まれている
   かつ未期限切れ かつ未失効 の場合、コンセント UI をスキップする
2. 新しいスコープが要求されたら、コンセント UI で「追加スコープのみ」明示し、同意取得
3. 同意は `granted_at + 365 日` で期限切れになる（GDPR の「定期的な同意更新」推奨に整合）
4. `prompt=consent` が指定されたら、既存同意があっても UI を再表示
5. ユーザーは `/end_session` 等のアカウント管理 UI から個別にコンセントを失効できる

## なぜ「クライアントごと」ではないか

- 後から `admin:write` のような強権限スコープが追加されたとき、過去の同意で自動許可されると
  原則違反になる
- GDPR の「特定の目的に対する同意」原則と整合させるため、scope 集合と紐づけて記録する

## なぜ「対話ごと」ではないか

- ユーザー体験が悪化する（毎リクエスト同意画面）
- それを避けるには「Remember me」のようなアドホックなフラグが必要になる

## 同意取り消しと既発行トークンの関係

コンセント取り消しは「以降の認可」のみに影響する。既発行のアクセストークン・
リフレッシュトークンの自動失効は **明示的に行う**（オプション）:

- 取り消しと同時に該当 client × sub のリフレッシュトークンを失効させる
- アクセストークンは TTL 内なら生きる（短寿命なので運用上問題ない）

これは ADR-004 のファミリー失効ロジックを再利用できる。

## 却下した代替案

- **コンセントを JWT として発行（持ち回り型）**: 取り消しが困難。サーバーサイドの
  失効リストが必要になり、結局ストアが要る
- **scope ごとに独立**: scope 数が増えるとレコード爆発・UI 複雑化
- **対話ごとに必ず再表示**: UX が悪く、ユーザーが「同意疲れ」を起こす

## 影響

- `consent.schema.json` は scope を配列として持つ（uniqueItems 制約あり）
- `requirements.md §2` がこの挙動を EARS で記述している
- `scenarios.feature` に「prompt=consent で再表示」と「既存同意でスキップ」の両方が含まれる
