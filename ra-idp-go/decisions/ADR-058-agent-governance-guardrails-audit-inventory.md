# ADR-058: エージェントガバナンス層 (ガードレール・委譲チェーン監査・インベントリ) を確定する

## ステータス

提案 (draft)。[[wi-59-agent-governance-guardrails-audit-inventory]] の意思決定を先行して
起草する。wi-59 の実装着手とともに「採用」へ移す。[[ADR-048]] (エージェント一級プリンシパル)・
[[ADR-049]] (token exchange による委譲と actor チェーン)・[[ADR-041]] (認証イベントモデル)・
[[ADR-018]] (監査 / アプリログ分離)・[[ADR-045]] (認証イベントの保持期間)・
[[ADR-029]] (ログイン試行スロットリング)・[[ADR-010]] (AuthZEN ポリシー)・
[[ADR-032]] (テナント一級集約) を前提に、エージェントの ID・委譲・認可の上に被せる
**運用統制 (governance) 層の最小核**を確定する。本 ADR は
[[wi-49-agent-identity-first-class-principal]]・[[wi-50-token-exchange-delegation-actor-chain]] で
導入したエージェント機能群に統制を被せる総仕上げにあたり、即時失効
([[wi-58-continuous-access-evaluation-agent-revocation]]) と協調する。

## コンテキスト

エージェントを第一級プリンシパル ([[ADR-048]]) として運用し、ユーザー代行と委譲チェーン
([[ADR-049]]) を許すと、ID・委譲・認可だけでは塞げない運用リスクが残る。自律・半自律な
エージェントは人間より高頻度・高並列に行動し、誤動作・暴走・プロンプトインジェクション時の
被害半径 (blast radius) が大きい。Microsoft Entra Agent ID / Agent 365、Okta・Ping の
エージェントガバナンスはいずれも、レート・予算上限、行動監査、ライフサイクル可視化を
統制の中核に据えている。

ra-idp-go には統制の素地が散在している。

1. **監査イベント基盤**: 認証イベントモデル ([[ADR-041]]) と監査 / アプリログ分離
   ([[ADR-018]])、保持期間と sweep ([[ADR-045]]) を既に持つ。
2. **委譲の来歴**: token exchange の `act` チェーンと委譲深さ ([[ADR-049]]) が
   「user → agent → subagent」を表現する。
3. **レート防御**: ログイン試行スロットリング ([[ADR-029]]) が認証経路の試行を遅らせる。

しかし、これらは**エージェント単位の統制**を欠く。すなわち、(a) エージェントごとの
レート・予算・行動回数・有効期限・許可リソースの上限、(b) `act` チェーンと委譲深さを
相関キーに据えた横断監査、(c) 全エージェントを所有者・最終活動・付与権限とともに棚卸す
インベントリ、が無い。本 ADR はこの 3 つを、既存基盤を**流用**して新設する境界を決める。
新たな監査ストアや課金エンジンを並行構築しない、という否定的境界も同時に確定する。

## 決定

1. **エージェント単位のガードレールを一級概念として導入する**。`AgentGuardrail` を新設し、
   1 つのエージェント ([[ADR-048]] の `Agent`) に 0 個以上束ねる。`GuardrailKind` は次の
   5 種別とする。
   - **rate**: 単位時間あたりのトークン発行 / 行動回数の上限。
   - **budget / cost**: コスト相当値の累積上限 (測定 HOOK のみ。決定 6 を参照)。
   - **action-count**: 一定期間または通算の行動回数の上限。
   - **expiry**: ガードレール / 委任の有効期限 (時刻到来で fail-closed に閉じる)。
   - **allowed-resources**: 行動可能な resource / audience の許可リスト
     ([[ADR-049]] の Resource Indicators と整合)。

2. **上限超過時の挙動はポリシーで選べる二段構えとし、既定は拒否 (fail-closed)**。
   ガードレール超過時の既定挙動は **deny by default** — 新規トークンを発行せず、行動を拒否する。
   テナント / エージェント単位の設定で、ハード拒否の代わりに**人間承認へ降格 (downgrade)** する
   policy を選べる。降格時は CIBA による非同期人間承認
   ([[wi-52-ciba-async-human-approval]]) をトリガーし、承認が得られた場合に限り続行する。
   いずれの policy でも、評価不能・タイムアウト・設定欠損は**拒否側**に倒す。降格は「拒否を
   緩める」のではなく「人間を介在させて承認を得る」経路であり、無承認の続行は起こさない。

3. **監査イベントは actor / 委譲チェーンと委譲深さを相関キーとして載せ、既存基盤を流用する**。
   ガードレール評価と行動の監査は、[[ADR-041]] の認証イベントモデルと [[ADR-018]] の監査ストアを
   **再利用**し、並行ストアを構築しない。各監査イベントは [[ADR-049]] の `act` チェーン
   (user → agent → subagent) と委譲深さ (delegation depth) を correlation key として保持し、
   「誰の委譲のもとで・どの深さで・どのエージェントが」行ったかを横断検索できる。新規イベントは
   `GuardrailConfigured` / `GuardrailBreached` / `AgentActionAudited` とし、既存 payload は
   拡張のみで破壊しない ([[ADR-041]] 後方互換方針)。

4. **エージェントインベントリのビューを設ける**。全エージェントを **所有者 (owner)**・
   **最終活動 (last activity)**・**付与権限 (granted permissions)** とともに一覧する
   `AgentActivitySummary` を新設する。所有者・kill-switch・status は [[ADR-048]] の `Agent`
   集約を典拠とし、付与権限は [[ADR-010]] の認可結果から導く。インベントリは新たな真実の源を
   作らず、既存集約とイベントの読み取りビューに徹する。

5. **ガードレールはトークン発行 / 行動経路で fail-closed に強制し、tenant-scoped とする**。
   評価点は [[ADR-048]] / [[ADR-049]] のトークン発行ゲート (client_credentials・token exchange) と
   行動経路に挟む。判定漏れ・評価器障害があっても「発行しない / 行動させない」側へ倒す
   (決定 2)。ガードレール・監査・インベントリはすべて tenant-scoped とし ([[ADR-032]])、
   cross-tenant な参照・適用を認めない。

6. **budget / cost は測定 HOOK に限定し、課金・計測エンジンを IdP 内に作らない**。
   budget / cost ガードレールは、外部の計測・課金基盤がコスト相当値を供給するための
   **HOOK (上限値の保持と上限超過の評価)** のみを提供する。トークン課金・metering・請求の
   集計エンジンは IdP の責務外とし、構築しない (決定 7 の却下案を参照)。HOOK が値を得られない
   間は、その budget ガードレールは未充足として扱い fail-closed する。

7. **保持・権限・レート統制の関係を確定する**。
   - **保持**: ガバナンス監査イベントの retention は既存方針 ([[ADR-045]]) に従い、
     新たな保持機構を作らない。
   - **権限**: ガードレール CRUD・エージェント別監査検索・インベントリ参照は新規 permission
     `AdminAgentGovernanceManage` で保護し、判定は [[ADR-010]] の `authorize()` 経由とする。
   - **レートとの関係**: ガードレールの rate は**エージェント単位の意図的な上限**であり、
     [[ADR-029]] のログイン試行スロットリング (攻撃 / 認証経路の防御) とは目的・スコープが
     異なる。両者は重複ではなく補完であり、エージェント rate は throttle の代替にしない。

## 影響

- 新規集約 `AgentGuardrail` と読み取りビュー `AgentActivitySummary` / `AgentAuditQuery` が
  加わる。いずれも [[ADR-048]] の `Agent`・[[ADR-049]] の actor チェーンを典拠とし、
  tenant_id 必須とする。
- トークン発行ゲート (client_credentials・token exchange) と行動経路にガードレール評価が挟まる。
  評価は fail-closed で、超過時は拒否または CIBA 降格 ([[wi-52]]) に分岐する。
- 監査イベント `GuardrailConfigured` / `GuardrailBreached` / `AgentActionAudited` が
  [[ADR-018]] の監査ストアへ流れ、[[ADR-041]] の検索 UI に actor チェーン・委譲深さの相関キー付きで現れる。
  retention は [[ADR-045]] に従う。
- 新規 permission `AdminAgentGovernanceManage` が RBAC / AuthZEN ([[ADR-010]]) に加わる。
- 管理 UI にエージェントインベントリ / 統制ダッシュボード、ガードレール設定、委譲チェーン込みの
  監査ビューが加わる。
- budget / cost は HOOK のみのため、外部計測基盤との接続インターフェースを定義するが、計測本体は
  含まない。

## 却下した代替案

- **ガバナンスを各アプリケーション層で実装する**: 各サービスが独自にレート・予算・回数を持つと、
  実装ごとに再生成され (re-generation risk)、上限・監査・降格挙動が不整合になる。統制はエージェントの
  ID・委譲・認可を握る IdP に属すべきであり、アプリ層へ分散させない。
- **IdP 内に課金・計測エンジンを構築する**: budget を厳密に評価しようとすると metering・請求集計まで
  抱え込み、IdP の責務が膨張する (scope creep)。コストは外部基盤が測り、IdP は上限保持と評価 HOOK のみ
  提供する (決定 6)。
- **監査を強くサンプリング / 集約して保存量を抑える**: per-action の `act` チェーンと委譲深さが失われると、
  「user → agent → subagent」の来歴が辿れず、エージェントの説明責任 (accountability) が崩れる。
  認証イベントの bucket 集約 ([[ADR-041]]) は計数の安全弁だが、委譲チェーンの個別保持を犠牲にしない。
- **エージェント単位の上限を設けない (ID・委譲・認可のみで足りるとする)**: 暴走したエージェントが
  上限なく発行・行動でき、被害半径が無制限になる。即時失効 ([[wi-58]]) は事後の止血であり、
  事前の上限 (fail-closed なガードレール) を代替しない。
- **ガードレール超過を常にハード拒否のみとする**: 拒否一択では、正当だが上限に達した業務が機械的に
  止まり運用が硬直する。fail-closed を既定としつつ、人間承認への降格 (CIBA, [[wi-52]]) を
  configurable な policy として用意し、安全性と運用継続性を両立させる。
