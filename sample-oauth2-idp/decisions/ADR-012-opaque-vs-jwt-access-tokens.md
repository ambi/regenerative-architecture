# ADR-012: アクセストークンを JWT（自己完結）として発行し、リフレッシュトークンは不透明（opaque）にする

## ステータス

採用

## コンテキスト

OAuth 2.0 でアクセストークンの形式は仕様上自由である。主な選択肢:

- **不透明（opaque）**: ランダム文字列。リソースサーバーは毎回 `/introspect` を呼ぶ
- **JWT (JWT Profile, RFC 9068)**: クレームを含む自己完結トークン。RS が単独で検証可能

各々のトレードオフ:

| 観点               | Opaque                          | JWT (RFC 9068)              |
|--------------------|---------------------------------|----------------------------|
| RS の検証コスト    | /introspect 必須（往復必要）   | JWKS だけで検証可能（軽量） |
| トークン失効       | 即時可能                       | TTL まで有効（または DB 突合） |
| クレームの可視性   | RS 側でゼロ知識               | クライアント・RS にクレーム見える |
| サイズ             | 数十バイト                     | 数百〜数千バイト           |
| センダー制約       | DB レコードで管理              | `cnf` クレームで自己完結    |

リフレッシュトークンの選択肢:

| 観点               | Opaque                          | JWT                          |
|--------------------|---------------------------------|-----------------------------|
| ローテーション     | DB レコード必須（自然）        | 状態管理が JWT 外に必要      |
| ファミリー失効     | DB クエリで効率的              | 失効リストが必要             |

## 決定

- **アクセストークン: JWT (RFC 9068)** をデフォルトとする
  - リソースサーバーが多い OIDC 環境では検証コストが支配的
  - JWKS で公開鍵を配布できれば RS は IdP に依存せず検証できる
  - `cnf.jkt` / `cnf.x5t#S256` でセンダー制約を自己完結的に表現できる
  - 短寿命（600 秒）にすることで「失効が即時にできない」リスクを許容範囲に収める

- **リフレッシュトークン: opaque** をデフォルトとする
  - ローテーションとファミリー失効が必須（ADR-004）であり、DB レコード前提のほうが自然
  - JWT で持つ意味がない（クライアント・RS が中身を見る必要がない）
  - 漏洩しても DB に `revoked=true` を立てれば即時失効できる
  - ストア側で SHA-256 ハッシュとして保存し、プレーン値は IdP に残さない

## なぜ Opaque アクセストークンを採用しないか

短寿命 JWT は失効ウィンドウのリスクを許容範囲に収められ、RS の検証パフォーマンスが
圧倒的に有利。特にマイクロサービスのように多数の RS がある環境では、毎リクエスト
`/introspect` を呼ぶのは現実的でない。

例外: **超高セキュリティ環境**（決済・医療）では opaque + introspection を選択する
余地がある。その場合は本 ADR を分岐させる新規 ADR（"ADR-012a: 決済 IdP プロファイルで
opaque アクセストークン採用"）を作成する。

## イントロスペクションエンドポイントの位置づけ

JWT を採用しても `/introspect` は提供する。理由:

- センダー制約付きトークンの `cnf` を確認したい RS のため
- リアルタイム失効状態を確認したい RS のため
- レガシー RS との互換性のため

ただし RS は **デフォルトで JWT 検証** を行い、`/introspect` は補助的に使う想定。

## 却下した代替案

- **両方 opaque**: 検証性能の問題で大規模デプロイに向かない
- **両方 JWT**: リフレッシュトークンの失効が困難
- **JWT 内に scope を入れずに opaque リファレンスを持つ**: 中途半端

## 影響

- `spec/tokens/access-token.schema.json` は JWT クレームを定義
- `spec/tokens/refresh-token.schema.json` は **ストアレコード** を定義（トークン文字列ではない）
- `adapters/persistence/` のリフレッシュトークンストアは SHA-256 ハッシュで保存する
- リソースサーバー実装者向けに「JWT 検証で十分」を Discovery と SDK ドキュメントで伝える
