- 確定日: 2026年6月14日
- 作成方法: 遡及復元

## 遡及復元であること

このベースラインは開発開始時に保存された原資料ではない。既存の SCL、ADR、README、
実装、テストから、失われていた上流意図を2026年6月14日に復元し、以後の比較基準として
確定したものである。過去の判断を新しい説明で置き換えないため、判断理由の詳細は各 ADR を正とする。

## 根拠

- `spec/scl.yaml`: 現在の規範仕様、標準採用範囲、モデル、状態、規則、SLO。
- `decisions/ADR-001` から `ADR-034`: 状態機械、OAuth/OIDC セキュリティ、永続化、
  監査、供給網、認証、管理、テナント分離に関する判断。
- `README.md`: システムの目的、対応プロトコル、ロードマップ、運用方法。
- `internal/spec/` と各テスト: SCL から実行可能な検証へ落とした範囲。
- `.github/workflows/ra-idp-go-ci.yaml`: 現在のCIで実行される検証。

## 確定した意図

1. ra-idp は OAuth 2.0 / OpenID Connect 対応 IdP であると同時に、
   Regenerative Architecture の実証対象である。
2. SCL をシステム振る舞いの単一の規範的上流ソースとする。
3. セキュリティ境界、標準採用範囲、非機能目標、保証義務を SCL に保持する。
4. 実装、アダプタ、UI、インフラは交換・再生成可能にする。
5. 重要な設計判断と、SCL だけから導出できない理由は ADR に保持する。
6. 高リスクな変更は、実装生成と独立した検証または決定的な検査を要求する。
7. ワークアイテムは SCL に限らず、ADR、障害、依存更新、運用要求からも作成できる。
8. ワークアイテムの completion には変更内容、対応関係、検証結果、未実施、例外承認、残リスクを残す。

## SCL への対応

| コンセプション上の意図 | SCL の主な対応先 |
| --- | --- |
| 標準準拠範囲 | `standards` |
| OAuth/OIDC の契約 | `interfaces`, `models` |
| トークンと認可のライフサイクル | `states`, `invariants` |
| 具体的な正常系・拒否系 | `scenarios` |
| Client・管理者・テナントの認可境界 | `permissions` |
| 性能、可用性、寿命、保持 | `objectives` |
| 合否基準と必要な検証 | `assurance` |

## 復元時に確認できなかった事項

- 開発開始時の依頼文、会話、優先順位の原文。
- 当初想定した利用組織、予算、納期、責任者。
- 初期版で意図的に対象外とした機能と、単に未着手だった機能の完全な区別。

これらは推測で補わない。将来の判断で必要になった時点で、コンセプションまたは ADR を更新する。

## 変更規則

- このベースラインの意図を変更する場合は、変更理由を ADR に残す。
- SCL と矛盾した場合は自動的にどちらかへ合わせず、意図変更か仕様欠落かを判断する。
- 新しいベースラインを確定した場合も、この文書は履歴として保持する。
