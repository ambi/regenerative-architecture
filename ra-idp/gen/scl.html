本仕様で使うドメイン用語の定義。PascalCase が正準名、aliases は実装上の別表記。

ResourceOwner

保護リソースに対するアクセス権限を付与できる実体。本アプリではエンドユーザー。

aliases resource_owner リソースオーナー EndUser
do not confuse with
  • Client — Client はアプリケーション、ResourceOwner は人間。OAuth では明確に区別される。
Client

ResourceOwner に代わって保護リソースへのアクセスを要求するアプリケーション。client_id で識別される。

aliases client クライアント RelyingParty
AuthorizationServer

ResourceOwner を認証し Client にトークンを発行するサーバー。本アプリそのもの。OIDC では IdP と同義。

aliases authorization_server IdP IdentityProvider
ResourceServer

AccessToken を提示する Client に対して保護リソースを返すサーバー。本アプリの外部。

aliases resource_server
Public

client_secret を安全に保持できない Client（SPA, ネイティブアプリ）。PKCE 必須。

aliases public
Confidential

サーバーサイドで client_secret または private_key を保持できる Client。

aliases confidential
AuthorizationCode

短命（60秒以下）の一度限り使用される中間トークン。/authorize で発行され /token で交換される。grant_type 値「authorization_code」と同じ文字列を共有する（RFC 6749 §4.1）。

aliases authorization_code 認可コード
RefreshToken

AccessToken の再発行に使うトークン。提示のたびにローテーションされる。絶対 TTL 30 日。grant_type 値「refresh_token」と同じ文字列を共有する（RFC 6749 §6）。

aliases refresh_token リフレッシュトークン
ClientCredentials

Client 自身が認可主体となる machine-to-machine フロー (RFC 6749 §4.4)。confidential のみ。

aliases client_credentials M2Mグラント
DeviceCode

入力制約のあるデバイス向けのデバイス認可グラント (RFC 8628)。grant_type は URN。

aliases urn:ietf:params:oauth:grant-type:device_code device_code デバイスコード
AccessToken

ResourceServer にアクセスする際に提示するトークン。JWT (PS256 / ES256) として発行、TTL 600秒。

aliases access_token アクセストークン
do not confuse with
  • IdToken — AccessToken は API アクセス用、IdToken は認証情報の表明用。混同してはならない。
IdToken

OIDC が定める、ResourceOwner の認証結果を表明する JWT。iss/sub/aud/exp/iat/auth_time/nonce/azp を含む。

aliases id_token IDトークン
Pkce

Proof Key for Code Exchange (RFC 7636)。code_challenge / code_verifier で認可コード横取り攻撃を防ぐ。本アプリでは例外なく必須。

aliases pkce PKCE
do not confuse with
  • Dpop — PKCE は認可コード保護、DPoP はトークン保護。攻撃面が異なる。
Dpop

Demonstrating Proof of Possession (RFC 9449)。JWK サムプリント (jkt) でトークンを所有鍵にバインドする。

aliases dpop DPoP
Mtls

Mutual-TLS Client Authentication (RFC 8705)。TLS クライアント証明書で Client を認証する。

aliases mtls mTLS tls_client_auth
Par

aliases par PAR

パスワードログイン後はセッション Cookie で認可を継続できる

ユーザー名・パスワード認証に成功すると HttpOnly セッション Cookie を発行し、同じ認可リクエストを継続する。

  1. 未認証セッションで 'web-app として認可リクエストを送る'
  2. browser login API に username alice と正しい password を送信する
  3. セッション Cookie が発行される
  4. 認可コードが redirect_uri に返る
  5. UserAuthenticated が発行される

browser login API の CSRF 不一致は拒否される

SameSite cookie と request token が一致しない browser login API は invalid_request。

  1. 未認証セッションで 'web-app として認可リクエストを送る'
  2. csrf 値を改ざんして browser login API を送信する
  3. エラー InvalidRequestError

prompt=login は既存セッションがあっても再認証を要求する

OIDC Core §3.1.2.1 — prompt=login は既存セッションの再利用を禁止する。

  1. ユーザー alice が現在時刻で認証済みである
  2. web-app として prompt login で認可リクエストを送る
  3. 認可リクエストの状態は AuthenticationPending

RP-Initiated Logout は登録済み post_logout_redirect_uri にだけリダイレクトする

OIDC RP-Initiated Logout 1.0 — logout 後の redirect URI はクライアント登録済み URI に限定する。

  1. confidential クライアント web-app を redirect_uri https://app.example.com/cb で登録する
  2. web-app として post_logout_redirect_uri https://app.example.com/cb でログアウトする
  3. state が post_logout_redirect_uri に伝播する

未登録 post_logout_redirect_uri は拒否される

open redirect を避けるため、logout redirect も登録済み URI のみ許可する。

  1. confidential クライアント web-app を redirect_uri https://app.example.com/cb で登録する
  2. web-app として post_logout_redirect_uri https://evil.example.com/cb でログアウトする
  3. エラー InvalidRequestError

client_credentials グラントで M2M トークンが発行される

RFC 6749 §4.4 — confidential クライアント自身のリソースアクセス。

  1. confidential クライアント backend を grant_types に client_credentials 含めて登録する
  2. backend として client_credentials で scope api:read のトークンを取得する
  3. 応答に access_token が含まれ refresh_token は含まれない
  4. 発行された access_token の sub は client_id と一致する
  5. AccessTokenIssued が発行される

public クライアントは client_credentials を登録できない

client_credentials は confidential 限定（TokenGrantClientCredentials 認可ルール）。

  1. public クライアント spa-app を grant_types に client_credentials 含めて登録する
  2. エラー InvalidRequestError

デバイス認可フローでアクセストークンを取得できる

RFC 8628 §3 — TV 等の入力制約デバイス向けの完全な正常系。

  1. confidential クライアント tv-app を grant_types に urn:ietf:params:oauth:grant-type:device_code 含めて登録する
  2. tv-app として scope openid profile でデバイス認可を開始する
  3. 応答に device_code・user_code・verification_uri・interval が含まれる
  4. ユーザー alice が verification_uri で user_code を入力し承認する
  5. device_code DC1 を交換する
  6. 応答に access_token と id_token が含まれる
  7. DeviceAuthorizationRequested が発行される
  8. DeviceAuthorizationApproved が発行される
  9. AccessTokenIssued が発行される

承認前のデバイス交換は authorization_pending を返す

RFC 8628 §3.5 — ユーザー承認前のポーリングは authorization_pending。

  1. AuthorizationPending 状態の device_code DC1 を用意する
  2. device_code DC1 を交換する
  3. エラー AuthorizationPendingError

ポーリング間隔より短い再試行は slow_down を返す

RFC 8628 §3.5 — interval 秒未満での再試行は slow_down で interval+5 秒に増やす。

  1. interval 5 秒の device_code DC1 を AuthorizationPending 状態で用意する
  2. device_code DC1 を交換する
  3. 2s 経過する
  4. device_code DC1 を交換する
  5. 2 回目はエラー SlowDownError

期限切れの device_code は expired_token を返す

RFC 8628 §3.5 — expires_in を超えた device_code 交換は expired_token。

  1. issued_at 2026-01-01T00:00:00Z・expires_at 2026-01-01T00:10:00Z の device_code DC1 を用意する
  2. 時刻が 2026-01-01T00:11:00Z になる
  3. device_code DC1 を交換する
  4. エラー ExpiredTokenError

private_key_jwt によるクライアント認証が成功する

RFC 7523 §2.2 — client_assertion の署名 JWT で認証成立。

  1. confidential クライアント fapi-app を token_endpoint_auth_method private_key_jwt・jwks 登録で用意する
  2. 認可コード AC1 を verifier v・有効な client_assertion で交換する
  3. 応答に access_token が含まれる

改ざんされた client_assertion は invalid_client で拒否される

  1. confidential クライアント fapi-app を token_endpoint_auth_method private_key_jwt・jwks 登録で用意する
  2. 認可コード AC1 を verifier v・改ざんされた client_assertion で交換する
  3. エラー InvalidClientError

tls_client_auth で mTLS の Subject DN が一致すれば認証成立

RFC 8705 §2.1.1。

  1. confidential クライアント mtls-app を token_endpoint_auth_method tls_client_auth・tls_client_auth_subject_dn CN=mtls-app で用意する
  2. 認可コード AC1 を verifier v・mTLS Subject DN CN=mtls-app で交換する
  3. 応答に access_token が含まれる

mTLS バインド AT は同じ証明書のリクエストでのみ受理される

RFC 8705 §3。cnf.x5t#S256 が AT に埋め込まれ、resource access では同一証明書の thumbprint 一致を要求する。

  1. confidential クライアント mtls-app を token_endpoint_auth_method tls_client_auth で用意する
  2. mTLS 証明書バインドされた access_token を発行する
  3. 同じ証明書を提示して userinfo を取得すると 200 を返す
  4. 別の証明書を提示すると invalid_token で拒否される

RFC 8414 メタデータ文書は OIDC Discovery と同等の内容を返す

同一のメタデータを別パスでも取得できる。

  1. Authorization Server メタデータを取得する
  2. 応答に issuer・authorization_endpoint・token_endpoint・jwks_uri・grant_types_supported が含まれる
