# Regenerative Architecture

*変化を前提とする時代のソフトウェア設計*

---

過去数十年間、我々は多くの設計上のアイデアを目にしてきた。

アーキテクチャ手法:

- Hexagonal Architecture (Alistair Cockburn)
- Onion Architecture (Jeffrey Palermo)
- Screaming Architecture
- DCI Architecture (James Coplien)
- BCE (Ivar Jacobson)
- Clean Architecture (Robert C. Martin)

これらは細部の違いこそあれ、すべて同じ目的を共有している。**関心の分離**である。フレームワークから独立し、テスト可能で、UIから独立し、データベースから独立し、外部エージェントから独立した設計を作ること。

開発手法:

- Domain-Driven Design (Eric Evans)
- Test-Driven Development (Kent Beck)
- Behavior-Driven Development (Dan North)
- Type-First Development
- Spec-Driven Development

これらが共通して追求してきたのは、**正しさと意図を構造として表現すること**である。ドメインの語彙を関係者で共有し、振る舞いをテストとして先に書き、型で不正状態を排除し、仕様を一級の成果物として扱う。

Regenerative Architecture はこれら両系統の遺産の上に立つ。しかし2020年代の半ばに入り、我々はこれらが想定していなかった新しい現実に直面している。

## 1 新しい現実

第一に、**AIによる攻撃の加速**。脆弱性発見とエクスプロイト生成のスピードは、防御側の修正・デプロイのスピードを容易に上回るようになった。「脆弱性を作らない」だけでは足りない。脆弱性が発見されてから本番に修正が届くまでの時間が、勝負を分ける。

第二に、**競争速度の加速**。AIによって誰でも速く作れる。先行優位は短くなり、追従できない者は退場する。

第三に、**依存先の変化の加速**。ライブラリ、ランタイム、OS、クラウドサービスはかつてないペースで変わる。追従し続けないと、気づけば技術的負債の山に埋もれている。

第四に、そしておそらく最も重要なのは、**AI自体の進化**である。半年前のAIで書かれたコードは、現在のAIから見れば最適ではない。来年のAIから見ればさらに劣後する。ソフトウェアは静的な資産ではなく、継続的に再生成され続ける動的なプロセスになりつつある。

この前提に立つなら、設計の目的は「一度きれいに作ること」ではない。**仕様を核にして、実装・境界・実行環境を繰り返し再生成できること**である。

## 2 中心原則: 再生成可能性

### 2.1 規則

Regenerative Architecture が従来のアーキテクチャ手法に加える、ただ一つの規則がある。

> **任意の外側の層は、その内側の層すべてが保持されていれば、自動的に再生成可能でなければならない。**

これは Clean Architecture の依存性の規則——外側は内側に依存し、内側は外側を知らない——をそのまま引き継いだ上で、もう一歩を要求する。「内側を守る」だけでなく、「内側が保たれれば外側を**作り直せる**」ことを設計の要件として明示する。この規則を満たすには、内側の層が外側の層を導出するのに十分な情報量を持たなければならない。

### 2.2 帰結: 何を保存するか

再生成可能性の規則は、古典的な問い「何が不変か」の意味を変える。

従来のClean Architectureは、ビジネスルール（エンティティ）が最も変わらず、フレームワークやUIが最も変わると考えた。これは2010年代の経験則としては正しかった。

しかし、AI時代において問うべきは「**何を保存しておけば、すべてを失っても再生成できるか**」である。実装は失われてもよい。フレームワークは捨てられてよい。コードベース全体が陳腐化しても構わない。ただし、**ある一定のものを保持していれば、新しいAI・新しい言語・新しいランタイムの上に、システムを再構築できる**。その「ある一定のもの」=「現実世界（ドメイン）を記述した仕様」を中心に据えるのが、Regenerative Architectureである。

層を分ける基準は、ビジネスルールかフレームワークかではなく、**再生成コストの非対称性**である。失われたときに復元が困難なものを内側に、容易なものを外側に置く。

### 2.3 帰結: 単一上流ソースと検証可能性

再生成が成り立つために、保存される仕様は二つの性質を満たさなければならない。

**単一上流ソース**: 同じ概念を複数形式で並行保守してはならない。契約を OpenAPI・JSON Schema・Protobuf に手で書き分ければ、その分だけドリフトが生まれ、どれが真実か分からなくなる。保存対象は単一の上流ソースに集約し、下流のインタフェース・スキーマは派生物として扱う。

**検証可能性**: 保存される仕様は機械実行可能でなければならない。自然言語の仕様は、AIが再生成するたびに解釈が揺れる。仕様が「再生されたコードの合否を判定できる形式」を取って初めて、再生はテスト可能になり、書き直しと区別される。

そして「不変なもの」自身も時間とともに変化する。ドメインの理解は深まり、用語は再定義され、状態機械は新たな遷移を要求される。各変更は次章で述べる決定記録（ADR）と対で進める。第1層が「何を保存するか」を、第2層が「なぜそう保存することにしたか」を定義し、両者は一体である。

## 3 5つの層

内側から順に、5つの層を置く。各層には「**保存対象**」と「**派生物**」の区別が貫かれる。保存対象は失われれば再生不能となるもの、派生物は保存対象から導出され、いつでも捨てて作り直せるものである。

### 3.1 第1層：Specification Core（仕様核）

最も内側。ここに置かれるものは、**機械実行可能かつ曖昧性のない形式**で表現された、システムの本質的な振る舞いである。特定のプログラミング言語に依存せず、かつ「自然言語で書かれたドキュメント」でもない。「実行すれば合否が判定できる形式」で記述される。なぜなら、自然言語の仕様は解釈で揺れ、AIが再生成するたびに微妙に振る舞いが変わるからだ。

#### 3.1.1 Specification Core Language (SCL)

第1層は **Specification Core Language (SCL)** によって記述される。SCL は実装言語・フレームワーク・データベース・ランタイムに依存せず、AI が解釈・生成可能で、人間が読め、長期保存可能な、単一の上流ソースである。仕様核に必要な多面性——契約・状態機械・行動仕様・不変条件・認可・非機能——を一つの整合した形式で保持し、多形式の手書き並行保守を排除する。

SCL は8つのセクションから成る。

- `vocabulary` — ユビキタス言語（用語・定義・別名・混同しやすい類義語）
- `models` — エンティティ・値オブジェクト・イベント・列挙・エラー
- `interfaces` — 外部との契約（入力・出力・エラー・発行イベント）
- `states` — 状態と遷移の宣言
- `invariants` — 普遍的に成り立つべき不変条件
- `scenarios` — 自然文ステップで書く受け入れ例
- `permissions` — 認可ルール（誰がどのリソースに何をできるか）
- `objectives` — SLO・性能・保持・ライフタイム・セキュリティの目標

各セクションの文法・プロパティ・型システム・式の文法は `SPECIFICATION_CORE_LANGUAGE.md` を参照のこと。

仕様核は SCL 本体——特定の言語に依存しないデータ——のみで構成される。「SCL の読み込み・型付け・検証コード」（TypeScript の Zod、Python の Pydantic など）は**仕様バインディング**で、仕様核ではなく**第3層（Application Logic）の最内側**に属する。目的言語が変われば書き直される生成物だからだ。

SCL は現時点の定義のみを保持し、バージョン間の変遷履歴は持たない。SCL の変更にともなうデータマイグレーション——旧スキーマから新スキーマへの変換ロジック・バックフィル・後方互換アダプタ——は、変更の意図を ADR（第2層）が記録し、新旧 SCL の差分から第3層・第4層が変換コードを導出・実行する。

#### 3.1.2 SCL からの派生

SCL は単一の上流ソースであり、下流の形式・コード・図・ルールはすべて SCL から派生する。

- インタフェース・スキーマ：OpenAPI / JSON Schema / Protobuf / AsyncAPI（`interfaces` `models` から）
- ポリシー・認可 API：Cedar / OPA Rego / Cerbos / AuthZEN（`permissions` `interfaces` から）
- 監視・SLO：OpenSLO・Prometheus アラート・負荷テスト合否（`objectives` から）
- 図：状態機械図・シーケンス図（`states` `scenarios` から）
- テスト：プロパティテスト・行動テスト（`invariants` `scenarios` から）
- 言語バインディング：Zod / Pydantic / Go 構造体（`models` から）

保存対象は SCL のみ。派生物はいつでも捨てて作り直せる。

---

> **保存対象**: SCL（vocabulary / models / interfaces / states / invariants / scenarios / permissions / objectives）
> **派生物**: 各インタフェース・スキーマ・ポリシー・認可 API・監視ルール・図・テスト・マイグレーションスクリプト・仕様バインディング

### 3.2 第2層：Decision Record（決定記録）

仕様だけでは、なぜその設計になったかは分からない。そこで、重要な決定とその理由を ADR に残す。

ADR に記録する内容:

- 採用した選択肢
- 捨てた選択肢
- 判断の前提
- 判断時点での制約
- 将来見直す条件
- 障害時の運用判断
- マイグレーションの知見
- フォールバック戦略
- リトライやタイムアウトの調整理由
- ポストモーテムやインシデントの学習結果

ADRは「書いて終わり」ではない。決定は覆る。したがって各ADRには、採用・廃止・置き換えのステータスと、置き換えた場合のリンクを明示する。

```text
## ステータス
廃止（ADR-015 "PostgreSQL全文検索への移行" に置き換えられた、2025-03-01）
```

廃止されたADRは削除しない。「なぜ一度そう決めたのか」という経緯も、将来の再生成において重要な文脈である。

> **保存対象**: ADRのテキスト群（採用・廃止履歴を含む）
> **派生物**: なし（第3-5層を再生成する際の制約・前提として参照される）

### 3.3 第3層：Application Logic（アプリケーション論理）

ここからが、従来のClean Architectureで言うところのユースケース層に近い。ただし強調点が異なる。

この層は、仕様核に定義された不変条件を満たす**手続き**を記述する。手続きは特定の言語で書かれるが、フレームワークには依存しない。

重要なのは、この層が**仕様核から導出可能**であるべきということだ。仕様核が十分に厳密であれば、この層はAIによって自動生成できる。手で書くこともできるが、それは仕様核との一致を継続的に検証されなければならない。

仕様バインディング（Zod・Pydantic 等、特定の目的言語で SCL を読み込む薄いレイヤ）も、この層の最内側に属する。

> **保存対象**: なし（層全体が派生物）
> **派生物**: 目的言語のユースケース実装・仕様バインディング

### 3.4 第4層：Adapter Layer（境界層）

外部世界との接続。データベース、HTTP、メッセージング、UI、外部API。

この層は、Hexagonal Architectureのポート＆アダプターの概念をそのまま借りる。Regenerative Architectureの観点では、この層は**最も積極的に捨てられる**べき層である。

> **保存対象**: なし
> **派生物**: DB接続・HTTPハンドラ・メッセージング・UI・外部API連携

### 3.5 第5層：Runtime & Infrastructure（実行環境とデリバリ）

最も外側。システムが**乗る土台**（OS、コンテナ、クラウドサービス、CDN）と、変化を**本番へ届ける経路**（デプロイパイプライン）。どちらも常に最新へ置き換えられる前提のコモディティであり、同心円の最外周に位置する。

デプロイパイプラインを構成するツール（CI/CD、デプロイ基盤）は、ランタイムと同じく差し替え可能なコモディティである。だからこそ第5層に属する。しかし、そのツールが**提供する能力**——「変化を速く・安全に本番へ届ける」——は、運用の都合ではなくアーキテクチャの要件である。なぜなら、良い仕様核と良い決定記録を持っていても、本番デプロイに2週間かかるなら、AIによる攻撃には対抗できないからだ。したがって、以下を実行環境の設計対象に含める。

- トランクベース開発
- Progressive delivery（canary, blue/green）
- Feature flag
- 自動ロールバック
- 契約テスト
- OpenTelemetryによる観測可能性
- SLSAに準拠したサプライチェーン保護
- Sigstoreによる成果物の署名

Nicole Forsgrenらの研究が示したように、**デプロイ頻度・変更のリードタイム・変更失敗率・MTTR**の4指標は、組織のソフトウェア能力を測る指標であり、Regenerative Architectureが実際に機能しているかの計測手段になる。

### 3.6 分割軸：境界づけられたコンテキスト

§3.1–3.5 の5層は**水平**の軸（技術的関心・抽象段階）である。機能が増えると、もう一つの**直交する垂直の軸**が要る。1つの機能変更が全層に散る（shotgun surgery）のを避け、変更を一か所に閉じ込めるための軸——**境界づけられたコンテキスト**である。

システムは (層 × コンテキスト) の格子になる。トップでコンテキストに分割し、各コンテキストの内部で5層を繰り返す。コンテキスト間は、相手の内部モデルではなく公開面のみを介して接続する。

この分割は再生成を弱めない。むしろ境界づけられたコンテキストは再生成の自然な単位であり、§2.2 の「再生成コストの非対称性」による判定が、モノリス全体ではなくコンテキスト単位で効くようになる。再生成・改修の粒度が細かくなるぶん安くなる。ただしコンテキスト境界の切り方は機械化できない人間の決定である（Conway の法則によりチーム構造を映しがちで、誤った境界は無分割より悪い）。
