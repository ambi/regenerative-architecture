# Regenerative Architecture

*変化を前提とする時代のソフトウェア設計*

---

過去数十年間、我々は多くの設計上のアイデアを目にしてきた。

アーキテクチャ手法:

- Hexagonal Architecture (Alistair Cockburn)
- Onion Architecture (Jeffrey Palermo)
- Screaming Architecture
- DCI Architecture (James Coplien)
- BCE (Ivar Jacobson)
- Clean Architecture (Robert C. Martin)

これらは細部の違いこそあれ、すべて同じ目的を共有している。**関心の分離**である。フレームワークから独立し、テスト可能で、UIから独立し、データベースから独立し、外部エージェントから独立した設計を作ること。

開発手法:

- Domain-Driven Design (Eric Evans)
- Test-Driven Development (Kent Beck)
- Behavior-Driven Development (Dan North)
- Type-First Development
- Spec-Driven Development

これらが共通して追求してきたのは、**正しさと意図を構造として表現すること**である。ドメインの語彙を関係者で共有し、振る舞いをテストとして先に書き、型で不正状態を排除し、仕様を一級の成果物として扱う。

Regenerative Architecture はこれら両系統の遺産の上に立つ。しかし2020年代の半ばに入り、我々はこれらが想定していなかった新しい現実に直面している。

## 1 新しい現実

第一に、**AIによる攻撃の加速**。脆弱性発見とエクスプロイト生成のスピードは、防御側の修正・デプロイのスピードを容易に上回るようになった。「脆弱性を作らない」だけでは足りない。脆弱性が発見されてから本番に修正が届くまでの時間が、勝負を分ける。

第二に、**競争速度の加速**。AIによって誰でも速く作れる。先行優位は短くなり、追従できない者は退場する。

第三に、**依存先の変化の加速**。ライブラリ、ランタイム、OS、クラウドサービスはかつてないペースで変わる。追従し続けないと、気づけば技術的負債の山に埋もれている。

第四に、そしておそらく最も重要なのは、**AI自体の進化**である。半年前のAIで書かれたコードは、現在のAIから見れば最適ではない。来年のAIから見ればさらに劣後する。ソフトウェアは静的な資産ではなく、継続的に再生成され続ける動的なプロセスになりつつある。

この前提に立つなら、設計の目的は「一度きれいに作ること」ではない。**仕様を核にして、実装・境界・実行環境を繰り返し再生成できること**である。

## 2 中心原則: 再生成可能性

### 2.1 規則

Regenerative Architecture が従来のアーキテクチャ手法に加える、ただ一つの規則がある。

> **任意の外側の層は、その内側の層すべてが保持されていれば、再生成可能でなければならない。**

これは Clean Architecture の依存性の規則——外側は内側に依存し、内側は外側を知らない——をそのまま引き継いだ上で、もう一歩を要求する。「内側を守る」だけでなく、「内側が保たれれば外側を**作り直せる**」ことを設計の要件として明示する。

### 2.2 保存対象と派生物

AI時代に問うべきは「**何を保存しておけば、実装を失っても再生成できるか**」である。実装、フレームワーク、ランタイムは捨てられてよいが、現実世界の要求を記述した仕様と、そこから導出できない判断は保存しなければならない。

層を分ける基準は、ビジネスルールかフレームワークかではなく、**機械的に導出できない情報か否か**である。失われたとき他の保存物から導出できないものを内側に、いつでも導出し直せるものを外側に置く。

### 2.3 単一上流ソースと検証

再生成が成り立つために、仕様と検証は次を満たさなければならない。

- **単一上流ソース**: 仕様は単一の上流ソースに集約し、下流のインタフェース・スキーマは派生物として扱う。
- **検証可能性**: 仕様は機械実行可能でなければならない。自然言語の仕様は、AIが再生成するたびに解釈が揺れる。仕様が「再生されたコードの合否を判定できる形式」を取って初めて、再生はテスト可能になり、書き直しと区別される。
- **追跡可能性**: 仕様、実装、テスト、検証結果を相互にたどれるようにし、仕様に根拠のない変更を残さない。
- **検証の独立性**: AIによる実装と同じ推論だけを合格根拠にせず、高リスクな変更には独立した実行や異なる手法による検証を求める。
- **未確認事項の明示**: 失敗した検証、未実施の検証、期限切れの検証結果、未解決事項を合格扱いせず、人間の承認または期限付きの例外承認を求める。

再生成は仕様固定下での**移植性**を保証する——同じ振る舞いを新しいランタイムや言語で再現する。AI の進化や依存先の刷新を取り込むのは、まずこの移植性によって可能になる。**振る舞い自身の改善**——ドメインの理解の深まりや要求の変化を反映する——は、移植性とは別の経路、すなわち仕様自身の進化を通じて行う。

## 3 5つの層

内側から順に、5つの層を置く。各層には「**保存対象**」と「**派生物**」の区別が貫かれる。保存対象は失われれば再生不能となるもの、派生物は保存対象から導出され、いつでも捨てて作り直せるものである。

### 3.1 第1層：Specification Core（仕様核）

最も内側。ここに置かれるものは、**機械実行可能かつ曖昧性のない形式**で表現された、システムの本質的な振る舞いである。特定のプログラミング言語に依存せず、かつ「自然言語で書かれたドキュメント」でもない。「実行すれば合否が判定できる形式」で記述される。なぜなら、自然言語の仕様は解釈で揺れ、AIが再生成するたびに微妙に振る舞いが変わるからだ。

#### 3.1.1 Specification Core Language (SCL)

第1層は **Specification Core Language (SCL)** によって記述される。SCL は実装言語・フレームワーク・データベース・ランタイムに依存せず、AI が解釈・生成可能で、人間が読め、長期保存可能な、単一の規範的上流ソースである。仕様核に必要な多面性——契約・状態機械・行動仕様・不変条件・認可・非機能・保証義務——を一つの整合した形式で保持し、多形式の手書き並行保守を排除する。

詳細は `SPECIFICATION_CORE_LANGUAGE.md` を参照のこと。

仕様核は SCL 本体——特定の言語に依存しないデータ——のみで構成される。「SCL の読み込み・型付け・検証コード」（TypeScript の Zod、Python の Pydantic など）は**仕様バインディング**で、仕様核ではなく**第3層（Application Logic）の最内側**に属する。目的言語が変われば書き直される生成物だからだ。

#### 3.1.2 SCL からの派生

SCL は単一の上流ソースであり、下流の形式・コード・図・ルールはすべて SCL から派生する。

- インタフェース・スキーマ：OpenAPI / JSON Schema / Protobuf / AsyncAPI（`interfaces` `models` から）
- ポリシー・認可 API：Cedar / OPA Rego / Cerbos / AuthZEN（`permissions` `interfaces` から）
- 監視・SLO：OpenSLO・Prometheus アラート・負荷テスト合否（`objectives` から）
- 図：状態機械図・シーケンス図（`states` `scenarios` から）
- テスト：プロパティテスト・受け入れ E2E テスト（`invariants` `scenarios` から）
- UI：Penpot / React / HTML / CSS (`user_experiences` `scenarios` から)
- 仕様バインディング：Zod / Pydantic / Go 構造体（`models` から）

仕様核としての保存対象は SCL のみ。派生物はいつでも捨てて作り直せる。

### 3.2 第2層：Decision Record（決定記録）

仕様だけでは、なぜその設計になったかは分からない。そこで、重要な決定とその理由を ADR に残す。

ADR に記録する内容:

- 採用した選択肢
- 捨てた選択肢
- 判断の前提
- 判断時点での制約
- 将来見直す条件
- 障害時の運用判断
- マイグレーションの知見
- フォールバック戦略
- リトライやタイムアウトの調整理由
- ポストモーテムやインシデントの学習結果

ADRは「書いて終わり」ではない。決定は覆る。したがって各ADRには、採用・廃止・置き換えのステータスと、置き換えた場合のリンクを明示する。

```text
## ステータス
廃止（ADR-015 "PostgreSQL全文検索への移行" に置き換えられた、2025-03-01）
```

廃止されたADRは削除しない。「なぜ一度そう決めたのか」という経緯も、将来の再生成において重要な文脈である。

コンセプション・ベースラインは規範仕様ではないが、人間由来の意図と SCL 生成時の前提を記録するため、この層で SCL の版に紐づけて保持する。

### 3.3 第3層：Application Logic（アプリケーション論理）

ここからが、従来のClean Architectureで言うところのユースケース層に近い。ただし強調点が異なる。

この層は、仕様核に定義された不変条件を満たす**手続き**を記述する。手続きは特定の言語で書かれるが、フレームワークには依存しない。

重要なのは、この層が**仕様核から導出可能**であるべきということだ。仕様核が十分に厳密であれば、この層はAIによって自動生成できる。手で書くこともできるが、それは仕様核との一致を継続的に検証されなければならない。

仕様バインディング（Zod・Pydantic 等、特定の目的言語で SCL を読み込む薄いレイヤ）も、この層の最内側に属する。

### 3.4 第4層：Adapter Layer（境界層）

外部世界との接続。データベース、HTTP、メッセージング、UI、外部API。

この層は、Hexagonal Architectureのポート＆アダプターの概念をそのまま借りる。Regenerative Architectureの観点では、この層は**最も積極的に捨てられる**べき層である。

### 3.5 第5層：Runtime & Infrastructure（実行環境とデリバリ）

最も外側。システムが**乗る土台**（OS、コンテナ、クラウドサービス、CDN）と、変化を**本番へ届ける経路**（デプロイパイプライン）。どちらも常に最新へ置き換えられる前提のコモディティであり、同心円の最外周に位置する。

デプロイパイプラインを構成するツール（CI/CD、デプロイ基盤）は、ランタイムと同じく差し替え可能なコモディティである。だからこそ第5層に属する。しかし、そのツールが**提供する能力**——「変化を速く・安全に本番へ届ける」——は、運用の都合ではなくアーキテクチャの要件である。なぜなら、良い仕様核と良い決定記録を持っていても、本番デプロイに2週間かかるなら、AIによる攻撃には対抗できないからだ。

CI は通常のビルド・テストに加えて、次を検証する。

- SCL とワークアイテムの構文、参照、未解決事項
- 仕様、実装、テスト、検証結果の双方向の対応関係
- 変更で影響を受ける保証義務の再評価と検証結果の有効期限
- 失敗、未実施、例外承認と、人間の承認条件との整合
- 生成物、依存物、実行結果の由来と改変の有無
- ワークアイテム内の完了記録と参照された機械出力の一致

リスク等級は、影響範囲、可逆性、機密性・完全性・可用性への影響、規制、データ移行、外部公開面から決める。高リスク変更ほど、独立した検証、段階配布、ロールバック演習、人間による仕様判断を強く要求する。低リスク変更は自動検査と、無作為に選んだ箇所のレビューだけで通してよい。

Nicole Forsgrenらの研究が示したように、**デプロイ頻度・変更のリードタイム・変更失敗率・MTTR**の4指標は、組織のソフトウェア能力を測る指標であり、Regenerative Architectureが実際に機能しているかの計測手段になる。

### 3.6 分割軸：境界づけられたコンテキスト

§3.1–3.5 の5層は**水平**の軸（技術的関心・抽象段階）である。機能が増えると、もう一つの**直交する垂直の軸**が要る。1つの機能変更が全層に散る（shotgun surgery）のを避け、変更を一か所に閉じ込めるための軸——**境界づけられたコンテキスト**である。

システムは (層 × コンテキスト) の格子になる。トップでコンテキストに分割し、各コンテキストの内部で5層を繰り返す。コンテキスト間は、相手の内部モデルではなく公開面のみを介して接続する。

この分割は再生成を弱めない。むしろ境界づけられたコンテキストは再生成の自然な単位であり、§2.2 の「機械的に導出できない情報か否か」による判定が、モノリス全体ではなくコンテキスト単位で効くようになる。再生成・改修の粒度が細かくなるぶん安くなる。ただしコンテキスト境界の切り方は機械化できない人間の決定である（Conway の法則によりチーム構造を映しがちで、誤った境界は無分割より悪い）。

## 4 コンセプションから検証まで

### 4.1 コンセプション

RA に則って開発するとき、人間は SCL を白紙から直接書かない。「どのような世界を実現し、どの損失を避けたいか」を AI に渡し、AI が質問、反例、選択肢を返しながら SCL を起草する。この最初の入力を**コンセプション（構想）**と呼ぶ。

コンセプションは SCL の代わりとなる規範仕様ではない。しかし、人間由来で再導出不能な意図と、AI が SCL をどう解釈したかを後から検査するため、承認された版を**コンセプション・ベースライン**として SCL の版に紐づけて保存する。SCL と矛盾するとき実装を直接規定するのは SCL だが、矛盾は自動的に上書きせず、SCL 修正または意図変更として人間へ差し戻す。

コンセプションは自由文だけでなく、少なくとも次の見出しを持つ。

1. **期待する成果**：利用者や現実世界に起こしたい観測可能な変化。
2. **主体と境界**：主体、信頼境界、含む範囲、明示的な対象外。
3. **必須事項と禁止事項**：守るべき鉄則と、してはならないこと。
4. **制約**：外部標準、法令、既存システム、規模、期限、予算。
5. **優先順位とトレードオフ**：何を優先し、何を犠牲にしてよいか。
6. **具体例と反例**：期待する例と、似ているが許されない例。
7. **未決定事項と仮定**：未決定事項、仮定、確信度、決定者。
8. **受け入れの指標**：成功・失敗を人間が観測できる指標。

AI は空欄を黙って補完しない。可逆で局所的な仮定は明示して起草できるが、認可、金銭、個人情報、データ破壊、外部契約、規制、不可逆な移行に関する未決定事項は、SCL の規範要件にする前に人間の承認を求める。

### 4.2 ワークアイテム

**ワークアイテム**は、一つの意味変更として説明・実装・検証できる作業単位である。SCL に限らず、ADR、障害報告、依存関係の更新要求などから作成される。少なくとも、根拠、対象範囲、範囲外、影響する保証義務、リスク等級、人間の承認が必要になる条件を持つ。AI はワークアイテムの範囲を越える変更を黙って行ってはならず、必要ならワークアイテムまたは仕様の変更を先に提案する。

ワークアイテムは状態を持つ。未完了の間は作業の根拠と予定検証を記録し、完了または中止の判断時には同じワークアイテムに**完了記録**を追記する。完了記録は、ワークアイテムの完了を判断するための記録であり、次を含む。

- 意味上の差分：利用者・運用者から見た変更点と変更していない点
- 対応関係：仕様から実装・テストへの対応と、実装変更から仕様・ADR・ワークアイテムへの逆参照
- 検証結果：対象版、手順、結果、環境、実行主体、成果物への参照
- 保証義務の状態：合格、失敗、未実施、例外承認
- 残リスク：影響、可能性、検出可能性、軽減策、見直し条件
- 人間の判断：仕様上の選択、例外承認、リスク受容

AI の文章による要約は索引であり、検証結果そのものではない。人間はワークアイテムの完了記録から、仕様差分、未充足、例外承認、高リスク箇所、無作為に選んだ箇所へレビューを絞る。すべてのコードを読むことを必須にはしないが、「軽く完全なレビュー」が可能だとも仮定しない。目的は、レビュー量をリスクに比例させ、未確認範囲を可視化することである。

完了記録は仕様上の真実ではないが、特定の変更を承認・配布した事実と、その時点の検証結果を保持する監査記録である。リリースや重要な変更に紐づけ、改変を検知できる形で保存する。

#### 4.2.1 保存規約

ワークアイテムの正本は、特定の課題管理サービスに依存しない、スキーマ検証可能なテキスト形式でソースリポジトリに保存する。既定の形式は YAML とし、各ワークアイテムに安定した一意識別子を付ける。

配置は ADR と同じく、対象となる境界づけられたコンテキストまたはコンポーネントの近くに置く。特定コンポーネントの作業はそのコンポーネント配下の `work-items/` に置き、リポジトリ全体の規約・横断ツール・複数コンポーネントにまたがる作業だけをルートの `work-items/` に置く。

```text
<component>/
  decisions/
    ADR-001-example.md
  work-items/
    wi-001-example.yaml
```

- `work-items/<work-item-id>.yaml` は作業開始前に作成し、根拠、対象範囲、範囲外、影響する保証義務、予定する検証、リスク等級、人間の承認条件を記録する。作業中はこのファイルだけがワークアイテムの正本である。
- `status` は `pending`、`in_progress`、`completed`、`cancelled` のいずれかとする。`completed` または `cancelled` にする時点で、同じワークアイテムファイルに `completion` セクションを追加する。
- `completion` は、完了日時または中止日時、意味上の差分、対応関係、実施した検証結果、保証義務の状態、残リスク、人間の判断を記録する。検証証跡の手順、実行環境、実行主体、対象ソース版、結果、証跡の保存先と要約値は、必要に応じて同じ `completion.evidence` に記録する。未完了の間は存在しなくてもよい。
- 大容量ログ、バイナリ、機密情報を含む証跡はワークアイテム YAML に直接埋め込まない。アクセス可能な保管先、内容の要約、改変検知用ハッシュを `completion.evidence[].artifacts` に記録する。小さなテキスト証跡をリポジトリに保存する場合は、ワークアイテムの正本と混同しない任意の artifacts 配下に置き、正本から参照する。

ワークアイテムと証跡は、SCL の版、ADR、ソースのコミット、生成物の要約値を安定した識別子で相互参照する。対話履歴や AI の最終応答だけを正本としてはならない。AI の応答は正本への索引として扱う。

GitHub 等を利用する場合、Issue はワークアイテム、Pull Request は変更と完了記録、CI Checks は検証結果、Review は人間の判断を表示・操作するアダプターとして使用できる。ただし外部サービス上の記録だけを正本とせず、対応する識別子と最終状態をリポジトリ内の正本に反映する。

### 4.3 開発フロー

開発フローは次の閉ループで行う。

1. **聞き出す**：AI がコンセプションを整理し、曖昧語、欠落、矛盾、反例、危険な仮定を質問として返す。
2. **基準を確定する**：人間が意図、優先順位、未解決事項、仮定の許容範囲を承認する。
3. **仕様化する**：AI が SCL と保証義務を生成し、各要素をコンセプション・ベースラインまたは ADR に追跡可能にする。
4. **反証を試みる**：別の文脈または手法で、反例、脅威、境界条件、仕様間矛盾を探索する。重大な未解決事項があれば実装へ進まない。
5. **作業に分ける**：AI が変更をワークアイテムに分割し、範囲、非目標、リスク等級、必要な検証、人間の承認条件を確定する。
6. **実装し検証する**：AI が実装し、CI が独立に再実行可能な検証結果を採取する。
7. **レビューし判断する**：人間は意味上の差分、仕様上の選択、未充足、例外承認、高リスク箇所、無作為に選んだ箇所をレビューする。
8. **運用から学ぶ**：本番観測、事故、誤検知を SCL、保証義務、テスト、ADR へ還流する。

マクロな指示だけで品質を保証する鍵は、AI の能力への信頼ではなく、この閉ループを途中で切らないことである。仕様に対応しない変更、検証結果に対応しない主張、決定者のいない未解決事項、期限のない例外承認は完了とみなさない。
